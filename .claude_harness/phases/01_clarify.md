# Phase 1: Clarify (Read-only)

## 목적
요구사항을 명확히 하고, UX/DB/아키텍처 방향을 확정한다.

## 권한
- **Read-only**: 코드 읽기, 문서 읽기만 허용
- 파일 생성/수정 금지 (산출물 `01_clarify_summary.md` 제외)

## 수행 절차

### 1. 서비스 블루프린트 확인
- `docs/SERVICE_BLUEPRINT.md`를 **반드시** 먼저 읽는다
- 서비스 정의, 핵심 가치, 사용자 여정을 파악한다

### 2. 디자인 시스템 확인
- 프론트엔드 관련 요구사항이면 `docs/design/DESIGN_SYSTEM.md`, `docs/design/design_tokens.json` 확인

### 3. 요구사항 분석
사용자의 기능 설명을 바탕으로 아래를 정리한다:

- **기능 요약**: 한 문장 정의
- **사용자 스토리**: "~로서, ~하고 싶다, ~를 위해"
- **Acceptance Criteria (AC)**: 검증 가능한 기준 목록 (AC 1-1, 1-2, ...)
- **범위 확정**: 포함/제외 항목 명시
- **UX 흐름**: 화면 전환, 사용자 인터랙션 요약
- **데이터 모델**: 필요한 엔티티, 필드, 관계
- **아키텍처 영향**: 기존 코드에 미치는 영향

### 4. 사용자 확인
- 정리한 내용을 사용자에게 보여주고 확인/수정 요청
- **사용자 승인 없이 다음 Phase로 넘어가지 않는다**

## 산출물
`specs/sessions/<session_id>/01_clarify_summary.md`

```markdown

## 기능 요약
...

## 사용자 스토리
...

## Acceptance Criteria
- AC 1-1: ...
- AC 1-2: ...

## 범위
### 포함
- ...
### 제외
- ...

## UX 흐름
...

## 데이터 모델
...

## 아키텍처 영향
...

## 사용자 확인
- [ ] 사용자 승인 완료
```

## 완료 조건
- 사용자가 AC 및 범위를 승인함
- `01_clarify_summary.md` 작성 완료
